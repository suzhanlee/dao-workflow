# 회의록 생성 가이드 (Generation Guide)

> Phase 5에서 사용하는 상세 생성 지침
> EBP(Evidence-Based Practice) 기반, 5단계 변환 프로세스

---

## 생성 원칙

### Core Practice 3가지

**1. 섹션 분리** — 결정 / 논의 / 액션을 반드시 분리 기록
- 결정(Decisions): 무엇으로 정해졌는가?
- 논의(Discussion): 어떻게 논의되었는가?
- 액션(Action Items): 누가 무엇을 해야 하는가?

**2. 행위자 명시 (AAA 트리플)**
| 요소 | 질문 | 예시 |
|------|------|------|
| Actor | 누가? | @철수 |
| Action | 무엇을? | 계획 수립 |
| Agreement | 언제까지? | MM-DD |

```
❌ "다음 주까지 계획 수립"
✅ "@철수가 MM-DD까지 계획 수립"
```

**3. 맥락 보강** — 결정의 이유, 논의의 배경을 반드시 포함
```
❌ 결정: A 방안 채택
✅ 결정: A 방안 채택 (이유: 비용 20% 절감 + 구현 기간 2주 단축)
```

## 5단계 변환 프로세스

### Step 1: 소스 분석 (Source Analysis)

전체 텍스트를 한 번에 읽으며 파악:
- 회의 주제/목적
- 참여자 식별 (이름/역할)
- 시간 정보 (날짜, 소요 시간)
- 회의 유형: 실험 설계 / 결과 분석 / 회고 / 계획 / 일반
- 핵심 질문: "이 회의의 핵심 질문은 무엇인가?" (1-3개)

### Step 2: 섹션 추출 (Section Extraction)

대화 내용을 결정/논의/액션으로 분류:

| 섹션 | 추출 키워드 |
|------|-------------|
| 결정 | 결정, 채택, 하기로, 확정 |
| 논의 | ~하는데, ~지만, 고려, 의견 |
| 액션 | ~하겠다, ~할게, 담당, 내가 |

중복/불필요한 내용 제거 후 각 섹션별 그룹화.

### Step 3: 구조화 (Structuring)

`template.md` 구조를 정확히 따라 각 섹션 채우기 (아래 섹션별 지침 참조).

### Step 4: 맥락 보강 (Context Enhancement)

| 섹션 | 보강할 맥락 |
|------|-------------|
| 결정 | 이유/배경, 합의 여부 |
| 논의 | 참여자 역할, 논의 흐름 (주장→반론→결론) |
| 액션 | 선행 조건, 의존 관계 |

### Step 5: 검토 (Review)

불참자 테스트 — 회의에 참여하지 않은 사람이 다음 질문에 답할 수 있는가?
1. 무엇이 결정되었는가?
2. 왜 그렇게 결정했는가?
3. 누가 무엇을 해야 하는가?
4. 다음 회의는 언제인가?

## 섹션별 채우기 지침

### 📌 기본 정보
- **날짜**: 오늘 날짜 (YYYY-MM-DD) 사용
- **시간**: 오디오 길이 기반 추정 (형식: HH:MM - HH:MM), 알 수 없으면 "미정"
- **참여자**: 텍스트에서 발견된 이름들, 없으면 "알 수 없음"
- **회의 유형**: 텍스트 내용 기반으로 분류
- **관련 실험 번호**: 텍스트에서 언급된 경우만 포함

### 🎯 회의 목표
한두 문장으로 이번 회의의 핵심 목표 정의. 텍스트의 서두나 회의 목적에서 추출.

### 📋 상황 맥락
이번 회의가 열리게 된 배경, 이전 실험 결과, 현재 프로젝트 상황 등.

### 🔍 가설 및 실험 설계
**관련 내용이 없는 경우**: "이번 회의와 관련된 가설 및 실험 설계 내용은 없습니다."
관련 내용이 있는 경우: 가설 → 실험 계획 표 → 주요 관찰/데이터 → 결과 분석 표 → 편향 및 제한사항 순서로 채우기.

### 💡 논의 내용
주제별로 분리하여 논의 요점 기록. 의견 차이가 있으면 명시.
주장-반론-결론 구조로 흐름 표현:
```
[발언자] 주장
   ↳ [발언자] 반론
      → 결론: ...
```

### 📚 Lesson Learn
텍스트에서 학습된 내용이 있는 경우만 포함:
- 연구 결과 Learn: 내용 + 적용 범위 + 데이터 근거
- 프로세스/방법론 Learn: 개선된 방법론
- 실패 Learn: 기대치 미충족에서 얻은 교훈

### 🚀 시스템/프로덕트 전환 고려
**관련 내용이 없는 경우**: "이번 회의에서 시스템/프로덕트 전환 관련 논의는 없었습니다."

### ✅ 결정사항
결정 내용 | 합의 여부(✅/⚠️/❌) | 다음 단계 형식으로 표 작성.
결정의 이유/배경을 반드시 포함할 것.

### 📝 Action Items
우선순위(🔴/🟡/🟢) | 할 일 | 담당자(@이름) | 기한(MM-DD) | 의존사항 | 완료 조건 형식.
- 담당자 없는 액션 → TBD로 표시
- 기한 없는 액션 → TBD로 표시

### 🔄 다음 회의 준비
- 다음 회의 일정: 텍스트에서 언급된 경우 MM-DD HH:MM, 없으면 "미정"
- 준비물: 참여자가 준비할 자료 체크리스트
- 다음 회의의 가설/질문

### 🔗 관련 문서 / 🏷️ 태그
텍스트에서 언급된 문서/링크/키워드 추출. 없으면 섹션 생략 가능.
